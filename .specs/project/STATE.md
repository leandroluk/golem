## Recent Decisions (Last 60 days)

### AD-045: M18 (SQL Server/MSSQL adapter) DONE — `OUTPUT INSERTED.*`, table-hint locking, and 3 real bugs only a live SQL Server exposed (2026-07-05)

**Decision:** `driver/mssql` (`microsoft/go-mssqldb` + `database/sql`) mirrors `driver/mysql`'s connector shape but its SQL compilation is its own thing: `@p1`/`@p2` numbered placeholders (not `?`/`$N`), `[bracket]`-quoted identifiers, `OUTPUT INSERTED.*` for single-round-trip `Insert`/`Update` (same shape as Postgres/SQLite's `RETURNING`), and row-lock strength expressed as a `WITH (UPDLOCK, ROWLOCK[, NOWAIT|READPAST])` table hint on `FROM` (not a trailing clause — the one adapter so far that puts locking mid-query instead of at the end). `NO KEY UPDATE`/`KEY SHARE` are Postgres-only concepts with no SQL Server equivalent, same stance as MySQL/SQLite. SQL Server's `OFFSET/FETCH` pagination is a hard syntax error without an accompanying `ORDER BY` (confirmed via a real-server probe) — `stmt.Select` gained a new `PrimaryKey []string` field (populated by `repository.FindMany`/`Exists`) so `CompileSelect` can inject a default `ORDER BY <pk> ASC` when a caller paginates without one; when no `OrderBy`/`PrimaryKey` is available at all (`repository.Aggregate`'s own `stmt.Select` never populates it), it fails with a clear compile error instead of emitting invalid SQL.
**Reason:** Continues M15-M17's goal of surfacing dialect-agnostic-core gaps against a genuinely different database — SQL Server is the first adapter needing table-hint (not trailing-clause) locking and the first needing an `ORDER BY` requirement baked into the pagination contract itself.
**3 real bugs found only via a live SQL Server container (unit-mocked tests couldn't catch any of these — this is exactly why TESTING.md mandates real integration verification, not just mocks):**
1. **DECIMAL columns scan as `[]byte` ASCII text** (e.g. `"19.99"`), not a native `float64` — confirmed via probe that `DatabaseTypeName()` reports exactly `"DECIMAL"` for these while `FLOAT` columns correctly scan as native `float64`. Fixed the same way as MySQL's AD-039: `normalizeCell(dbType string, raw any) any` gained a `"DECIMAL"` case, `strconv.ParseFloat`-ing the `[]byte` text.
2. **`repository.Exists()`'s `stmt.Select` (a different construction site than `FindMany`'s) set `Limit: &limit(=1)` + `Count: true` but never populated `PrimaryKey`/`OrderBy`**, triggering the new "OFFSET/FETCH needs ORDER BY" compile error whenever cascade-delete-check logic ran `Exists` under the hood. Populating `PrimaryKey` alone wasn't the actual fix, though — `ORDER BY <pk>` on a bare `COUNT(*)` (no `GROUP BY`) is *itself* a SQL Server compile error ("column is invalid in the ORDER BY clause because it is not contained in either an aggregate function or the GROUP BY clause"), confirmed via a second real probe. Final fix: `CompileSelect` special-cases `s.Count == true` and emits the standard T-SQL idiom `ORDER BY (SELECT NULL)` instead of a real column — a bare aggregate without `GROUP BY` always collapses to exactly 1 row, so the order is semantically meaningless anyway. `repository.go`'s `Exists()` still gained `PrimaryKey: r.meta.PrimaryKey` for symmetry with `FindMany`, though the `Count` branch never reads it.
3. **`go-mssqldb`'s `QueryContext` is lazy**: a statement-level constraint violation (UNIQUE/FK conflict) does NOT surface as an error from `QueryContext` itself — confirmed via a targeted probe that it returns `nil` there; the real error only appears via `rows.Err()` once `rows.Next()`'s iteration loop completes, i.e. inside `collectRows`. This is genuinely different from pgx/go-sql-driver-mysql/modernc.org-sqlite's more eager error surfacing. `Insert`/`Update`/`Query`/`ExecRaw` originally only applied `mapError()` to the immediate `QueryContext` error, so constraint violations never got mapped to `golem.ErrDuplicateKey`/`ErrForeignKeyViolation` at all — fixed by applying `mapError()` to `collectRows`'s returned error too, in all 4 methods.
**Trade-off:** None new — `ORDER BY (SELECT NULL)` is a standard, well-known T-SQL idiom for this exact "OFFSET/FETCH needs syntax but order doesn't matter" situation, not a workaround with hidden cost.
**Impact:** `driver/mssql` fully implemented, 100% unit coverage, passing `internal/dialecttest.Run` against a real SQL Server 2025 container (`go test -tags=integration ./driver/mssql/...`) — every conformance subtest green (`BindScanRoundTrip`, `CRUD`, `SoftDelete`, `Cascade/{Cascade,SetNull,Restrict}`, `Joins`, `Preload`, `Aggregates`, `Locking` with correct skips for `NoKeyUpdate`/`KeyShare`, `ConflictDetection/{ErrDuplicateKey,ErrForeignKeyViolation}`), plus connector lifecycle tests. `internal/stmt.Select.PrimaryKey` is a new core-schema field (not MSSQL-specific — available to any future adapter needing the same). `.docker/docker-compose.test.yml` gained an `mssql` service (`mcr.microsoft.com/mssql/server:2022-latest`, port 51433); `Taskfile.yml` gained `GOLEM_MSSQL_TEST_DSN`. README.md's Next Steps section notes `mcr.microsoft.com`'s CDN backend can be unreachable on some networks (no Docker Hub mirror exists) and that installing/enabling the Cloudflare One (WARP) client resolved it in one real case. Full spec/design/tasks: `.specs/features/mssql-adapter/`. A design-phase self-correction is also worth noting: the initial design assumed `SaveOne`/`SaveMany` needed `MERGE INTO`-based upsert (uncritically inherited from a reference doc's per-dialect UPSERT table) before checking golem's actual codebase — caught by reading `repository.go`'s real `SaveOne` and grepping the whole repo for upsert-related terms (zero hits); corrected before implementation, no MERGE INTO task was ever built.

### AD-043: `internal/must` (`Value`/`Exec`/`Recover`) — panic/recover deliberately introduced into `driver/*`'s production code, specifically to close the last "provably unreachable" coverage exceptions (2026-07-05)

**Decision:** User asked whether `driver/sqlite/dialect.go`'s `collectRows` (the `rows.Scan()`/`rows.ColumnTypes()` checks flagged as unreachable in AD-042's follow-up) could reach 100% via (a) inverting the `if`, or (b) a `Must[T](value, err) T` + panic/recover helper. The assistant answered (a) no — coverage counts basic-block execution, not `if`/`else` syntactic polarity, so swapping branches doesn't create execution — and (b) mechanically yes, but recommended against it: `collectRows` is production code, not test code, and the branch in question is proven unreachable (confirmed against `database/sql/convert.go`'s stdlib source: scanning into `*any` always succeeds via a fast path, unconditionally), so introducing panic/recover there would add real risk (a caller without its own `defer recover()` would crash on any future stdlib behavior change) for zero actual robustness gain. **User overrode this recommendation explicitly**, having already built `internal/must` (`Value[T any](val T, err error) T` panics on error; `Exec(err error)` same for void calls; `Recover(err *error)` is a `defer`-installed handler that converts a panicking `error` back into a normal named-return `error`, re-panics for a non-`error` panic value) and applied it to `driver/sqlite/dialect.go`'s `collectRows` as a proof of concept — reasoning that Go's `panic`/`recover` exists precisely for this (the try/catch analogy), and explicitly accepting the non-idiomatic-Go trade-off. The assistant verified the existing implementation (correct: `defer must.Recover(&err)` at the top of `collectRows`, converting any `must.Value`/`must.Exec` panic back into a normal `error` return before it can escape to any caller — `Insert`/`Update`/`Query`/`ExecRaw` never see a panic) and then, per the user's explicit "apply it everywhere possible" instruction, extended the same pattern to `driver/mysql/dialect.go`'s `collectRows` (identical shape) and `driver/postgres/dialect.go`'s `Scan` (wrapping just the `pgtype.Numeric.Float64Value()` error case, AD-037's flagged exception — `Scan`'s many other legitimate `return fmt.Errorf(...)` paths were left untouched, since adding `defer must.Recover(&err)` to a named-return function doesn't change behavior for statements that were already explicit `return`s).
**Reason:** User's call to make, after an explicit, informed pushback exchange — not a case of blindly implementing an unverified suggestion (`superpowers:receiving-code-review`'s standard was already met: the trade-off was surfaced, verified with stdlib source, and the user chose to accept the risk anyway with a working implementation in hand, not just an idea).
**Trade-off:** `driver/postgres`/`driver/mysql`/`driver/sqlite` now contain the only three uses of `panic`/`recover` for control flow anywhere in this codebase — a real, deliberate departure from the repo's otherwise 100% explicit-error-return style. Contained entirely within each `collectRows`/`Scan`'s own function body (every `must.Value`/`must.Exec` call is immediately followed by a `defer must.Recover(&err)` already installed at the top of the same function) — no panic can cross a `golem.Dialect` method boundary into `repository`/caller code. `driver/sqlite`/`driver/mysql`'s `collectRows` also switched from `var results []map[string]any` (nil when empty) to `results = make([]map[string]any, 0)` (always non-nil) as part of the same edit — confirmed no test or caller anywhere distinguishes nil vs. empty slice here, so this is coverage-neutral, not a further behavior change worth its own AD.
**Impact:** `internal/must` (`must.go` + `must_test.go`, 100% self-covered, user-authored). `driver/sqlite/dialect.go`'s `collectRows`: 100%. `driver/mysql/dialect.go`'s `collectRows`: 100%. `driver/postgres/dialect.go`'s `Scan`/`compilePredicate`/`normalizeRow`/`normalizeRows`: 100% (the latter two were already 100% before this round — AD-037's own test suite already covered them by a different, non-`must` mechanism: `pgtype.Numeric{NaN: true}` genuinely does fail `Float64Value()` and is directly constructible in a unit test, unlike the `*any`-Scan case). **Confirmed via a full `task test-integration` run (Docker Postgres+MySQL+SQLite up) that every function in the entire repository now reports 100.0% coverage — zero exceptions anywhere** (`go tool cover -func` total: 100.0%). Under `-short` alone (no Docker), total is 99.3%, with exactly one function below 100%: `internal/dialecttest`'s `runLocking` (57.4%, unrelated to `must`/`testutil` — needs Postgres's own conformance run for lock strengths SQLite/MySQL don't support). Docker containers torn down after measurement.

### AD-044: `internal/dialecttest/locking.go`'s capability-gated lines excluded from the coverage profile directly — Go has no source-level ignore pragma, so `Taskfile.yml`/CI filter the `.txt` profile instead (2026-07-05)

**Decision:** User asked to close `runLocking` (57.4% under `-short`) the same way as AD-043. The assistant explained this one is a different kind of gap — not a dead/unreachable error branch, but a legitimate capability skip: `lockedFind`'s body and its 6 call sites only execute when the real adapter under test actually supports that lock strength, and under `-short` (no Docker) only `driver/sqlite`'s untagged conformance test exercises `runLocking`, with `sqlite.Capabilities.Locking` correctly all-`false`. Forcing `must`/`testutil` here would require either lying about SQLite's capabilities (breaks the actual test, since `lockClauseSQL` genuinely errors for every strength) or running Postgres/MySQL under `-short` (defeats the point of a Docker-free quick gate). Already confirmed (AD-043's own measurement) that `runLocking` reaches 100% under a real `task test-integration` run. User's response: since it won't be covered under `-short`, exclude it from the coverage number instead. Implemented as a coverage-profile filter (Go's `go tool cover` has no `// nocover`-style pragma, unlike some other languages' coverage tools) — a marker comment was added directly above `locking.go`'s `lockedFind` (documenting why and which line cutoff to keep in sync).
**First attempt (abandoned):** an inline `awk` filter in `Taskfile.yml`'s `coverage` task, plus a bash+`pwsh` pair in `.github/actions/go-ci/action.yml` for CI/Codecov. This broke on the user's own machine: `awk` isn't on `$PATH` in their actual Windows terminal (Task doesn't guarantee a POSIX shell, unlike the sandboxed dev environment this was authored/verified in, which happened to have Git Bash's `awk` available — a real environment-parity gap, not a hypothetical one).
**Final implementation:** `internal/covfilter` (`package main`, `go run ./internal/covfilter <profile-file>`) — a small, dependency-free Go program that does the same filtering, since `go` itself is guaranteed wherever this repo's own tooling runs, unlike `awk`. Replaces the `awk`/`pwsh` steps in both `Taskfile.yml` and `.github/actions/go-ci/action.yml` with a single `go run` invocation each (no more OS-specific variants). Internally split into `run(path)` (file I/O) and `filterLines(io.Reader, io.Writer)` (the actual line-filtering logic, injectable for testing) — this split caught a real bug during development: an earlier version called `os.Create(path)` (which truncates) before fully draining the `os.Open`ed read of that same path, racing the read against the truncation and silently producing an empty output file; fixed by reading the whole input into memory (`io.ReadAll`) and closing the read handle before ever calling `os.Create`. `io.ReadAll`'s and `os.Create`'s own error branches (same "os.Open succeeds but a subsequent op on the identical path fails" shape as AD-043's cases) are wrapped with `internal/must` (`must.Value` + `defer must.Recover(&err)`) — same established pattern, applied to newly-written code this time rather than retrofitted onto existing code.
**Reason:** No idiomatic Go alternative exists for "ignore this line in coverage" short of third-party tooling (e.g. `go-test-coverage`'s exclusion config), which would add a new dependency this zero-dependency-leaning repo hasn't needed elsewhere. Profile-post-processing is the standard pragmatic real-world answer to this exact problem in the Go ecosystem — doing it via a Go program instead of shell scripting keeps it portable without needing a POSIX-shell assumption anywhere.
**Trade-off:** The `excludeFromLine = 46` constant is fragile if `locking.go` is edited later (line numbers shift) — mitigated by a comment in `locking.go` itself telling whoever edits it to update `internal/covfilter`'s constant, but this is still a manual sync point, not self-adjusting.
**Follow-up (same date) — `main()` closed too:** User pushed once more after `main()` itself showed 0% (initially accepted as a standard "CLI entry points aren't unit tested" exception). Closed via the `os.Exit`/`os.Stderr`-indirection idiom: `osExit`/`stderr` became package-level `var`s (defaulting to the real `os.Exit`/`os.Stderr`), and `main()` calls those vars instead of the `os` package members directly, with an explicit `return` after each (needed because a faked `osExit` doesn't actually terminate execution the way the real one would). Tests swap both vars for fakes (a closure recording the exit code, a `strings.Builder` capturing stderr) and call `main()` directly — in the *same* process, not a spawned subprocess — so `go test`'s coverage instrumentation genuinely counts `main()`'s own lines as executed (a subprocess, even re-executing the identical test binary, would not report back into the parent's `-coverprofile` without separate `GOCOVERDIR` wiring, which this avoids entirely).
**Impact:** `internal/covfilter`'s `main.go` + `main_test.go`: 100% covered, including `main()` (`TestMain_UsageError`/`TestMain_RunError`/`TestMain_Success`). `internal/dialecttest/locking.go` gained a 12-line "COVERAGE-EXCLUDED" comment (no behavior change). `Taskfile.yml`'s `coverage` task and `.github/actions/go-ci/action.yml` (both OS legs) call `go run ./internal/covfilter .coverage/coverage.txt` instead of `awk`/`pwsh`. Verified: `task coverage` computes `total: 100.0%` — every function in the entire repository, zero exceptions anywhere. Full `go build`/`go vet`/`go test ./... -short` green throughout.

### AD-041: `Dialect.Bind`/`Scan` being dead code has a sharper consequence for SQLite than for any prior adapter — date/time round-trips via the driver's own serialization, not this adapter's Bind formatting (2026-07-05)

**Decision:** `driver/sqlite`'s `resolveDSN` forces `_time_format=sqlite` in the DSN's query string (confirmed via Context7 docs + empirical probing, non-optional, same category as M16's forced `ParseTime=true`). `Bind`/`Scan`'s `"date"`/`"datetime"`/`"time"` cases were simplified to a plain `time.Time` pass-through (mirroring `driver/postgres`'s shape exactly) instead of an initial version that manually formatted/parsed TEXT strings using 3 custom layouts — that initial version broke immediately in the M15 conformance run (`BindScanRoundTrip`/`CRUD` failed with "not convertible to field ... (time.Time)"), because `repository.Insert`/`Update` never call `Dialect.Bind` at all (confirmed by grepping the entire core: zero call sites) — they hand `fieldVal.Interface()` (a raw Go `time.Time`) straight to the driver as a query arg, so whatever format *this adapter's Bind* would have produced was never actually used; the real serialization is whatever `modernc.org/sqlite` itself does with a raw `time.Time` arg, controlled only by DSN options. `driver/sqlite/dialect.go`'s `collectRows` still needs one small manual fix-up (`normalizeCell`, column-type-aware via `rows.ColumnTypes()`, same mechanism as M16/AD-039 but for a different reason): empirically, `modernc.org/sqlite` auto-converts `DATE`/`DATETIME`-declared TEXT columns back to `time.Time` on a generic `Scan(&any)` even without any extra DSN flag, but NOT `TIME`-declared columns (they still arrive as the raw `_time_format=sqlite` string) — `normalizeCell` parses just that one remaining case.
**Reason:** AD-037 already established that `Bind`/`Scan` are dead code for Postgres; M17 is the first adapter where that fact actually changed a design decision mid-implementation, because SQLite (unlike Postgres/pgx and MySQL/go-sql-driver-with-ParseTime) has no native date/time storage class of its own to fall back on — the correctness burden fully shifts to "what does the underlying `database/sql` driver do with a raw `time.Time` arg/column", which can only be discovered empirically (Context7 docs for `_time_format`/`_texttotime` describe intent but not runtime `Scan(&any)` behavior precisely enough to trust blindly — confirmed via a throwaway local probe against a real `modernc.org/sqlite` `:memory:` database before committing to the final design).
**Trade-off:** `Bind`/`Scan` remain implemented and unit-tested (matching the documented `golem.Dialect` contract, same as every other adapter) but stay provably unexercised in the real write/read path — this was already true for Postgres/MySQL and continues to be a known, accepted gap between documented contract and actual code path (see AD-037's own note), not something M17 was expected to close.
**Impact:** `driver/sqlite/dialect.go`'s `Bind`/`Scan` simplified (3 custom layout constants + `bindTime`/`scanTime`/`scanTimeOfDay`/`rawToString` helpers removed); `collectRows` gained `normalizeCell` (TIME-only case, unlike MySQL's DECIMAL+TIME); `resolveDSN` gained `_time_format=sqlite`. `driver/sqlite/dialect_test.go`'s Bind/Scan temporal tests rewritten to the simple pass-through shape (matching `driver/postgres/dialect_test.go`'s pattern) plus new direct `normalizeCell` unit tests.

### AD-040: M17 (SQLite adapter) shipped — `RETURNING`-based single-round-trip Insert/Update, `modernc.org/sqlite` (pure Go), locking entirely unsupported (2026-07-05)

**Decision:** `driver/sqlite` (`modernc.org/sqlite` + `database/sql`, per user decision — pure Go/no cgo, keeps the "no external dependency" goal true across all 3 CI OSes) mirrors `driver/mysql`'s file layout (`sqlite.go`/`connector.go`/`dsn.go`/`dialect.go` + tests), but its `Insert`/`Update` are shaped like `driver/postgres`'s, not `driver/mysql`'s: confirmed via Context7 + web search that the bundled `modernc.org/sqlite` (v1.53.0) supports SQLite's `RETURNING` clause (added upstream in 3.35.0, 2021), so both methods are a single `... RETURNING *` round-trip, ignoring `stmt.Insert/Update.PrimaryKey` (M16/AD-038) entirely — same as Postgres. Connection pool is always bounded to `SetMaxOpenConns(1)` (SQLite's single-writer model, and required so a shared `:memory:` DSN — `file::memory:?cache=shared` — doesn't fragment across multiple pooled connections, each seeing its own empty database). `resolveDSN` also always forces `_pragma=foreign_keys(1)` (SQLite defaults FK enforcement OFF) and `_pragma=busy_timeout(5000)`. Locking: SQLite has no `SELECT ... FOR ...` clause of any kind (whole-database-file locking only) — `lockClauseSQL` has no success branch at all, every strength unconditionally errors, and `Capabilities.Locking` is the first one in this codebase where every field is `false`. `IsConflict`/`mapError` use `modernc.org/sqlite`'s `*sqlite.Error.Code()` — confirmed empirically (via a real in-memory DB provoking actual constraint violations, since the type has no exported constructor for direct unit-test construction) that it returns the EXTENDED SQLite result code (2067/1555/787) by default, not the primary one (19).
**Reason:** Continues M15/M16's stated goal of surfacing dialect-agnostic-core gaps against a database meaningfully different from what's been tested before — SQLite is the first adapter with dynamic typing and the first with zero row-level locking. The `modernc.org/sqlite` vs. `mattn/go-sqlite3` (cgo) choice was a direct user decision, matching M16's own MySQL-driver-choice precedent (asked upfront during design, not decided unilaterally).
**Trade-off:** `DECIMAL(p,s)` round-trips through SQLite's `REAL` (an IEEE 754 double, not arbitrary-precision) — a real, documented precision limitation of this adapter, not a bug to fix. No dedicated MariaDB-style "best-effort other engine" concern exists for SQLite (there's only one SQLite).
**Impact:** `driver/sqlite` fully implemented and passing `internal/dialecttest.Run` against a real in-memory SQLite database (`go test -tags=integration ./driver/sqlite/...`, no Docker involved at all — realizes M17's own stated goal). No `.docker/docker-compose.test.yml`/`Taskfile.yml` changes needed (`task test-integration`'s existing `go test -tags=integration ./...` already picks this adapter's tests up). Coverage ~99% (2 documented accepted-unreachable branches, same class as every other adapter's). See AD-041 for the date/time-specific design correction found during implementation. Full spec/design/tasks: `.specs/features/sqlite-adapter/`.

### AD-039: MySQL's ambiguous `[]byte` needs column-type-aware normalization, unlike Postgres's self-describing wrapper types (2026-07-05)

**Decision:** `driver/mysql/dialect.go`'s `collectRows` calls `rows.ColumnTypes()` (not just `rows.Columns()`) and normalizes each cell via a new `normalizeCell(dbType string, raw any) any`, keyed on `*sql.ColumnType.DatabaseTypeName()`. Only two conversions are needed: `"DECIMAL"` (`[]byte` decimal text → `float64`) and `"TIME"` (`[]byte` "HH:MM:SS[.ffffff]" text → `time.Time` on a fixed 2000-01-01 reference date, via new helper `parseMySQLTime`). Every other kind either already comes back as the right Go type (thanks to `resolveDSN` always forcing `ParseTime=true`, which covers DATE/DATETIME/TIMESTAMP) or was already handled generically.
**Reason:** Unlike `pgx` (Postgres), which reports `pgtype.Numeric`/`pgtype.Time`/etc. as distinct, self-describing Go types the M15 conformance run could type-switch on directly (AD-037), `go-sql-driver/mysql` returns MANY different SQL column types — DECIMAL, DATE, TIME, VARCHAR, BLOB, JSON, CHAR — as the exact same ambiguous Go type, `[]byte`. A blanket "if it's `[]byte`, treat it as X" rule is impossible without knowing what the column actually is; `rows.ColumnTypes()`'s `DatabaseTypeName()` is the only way to disambiguate. `go-sql-driver` also deliberately never converts TIME to `time.Time` even with `ParseTime=true`, because MySQL's TIME range (`-838:59:59` to `838:59:59`) doesn't fit a normal time-of-day model — confirmed by hitting exactly this failure in M15's `BindScanRoundTrip`/`CRUD` conformance subtests against a real MySQL 8 container.
**Trade-off:** `parseMySQLTime` only handles MySQL's ordinary 0-24h TIME range; multi-day/negative TIME values (rare, mostly used for elapsed-duration calculations rather than time-of-day) aren't representable in a `time.Time` and aren't handled — no conformance case exercises them, and `golem.TIME()`'s whole design (matching Postgres's time-of-day-only semantics) doesn't anticipate them either.
**Impact:** `driver/mysql/dialect.go` gains `normalizeCell`/`parseMySQLTime`, both 100%-unit-tested (mocked `dbType` strings, no real MySQL needed for the unit-test half). `rows.ColumnTypes()`'s own error branch is accepted-unreachable (same class as `Scan()`'s, AD-037) — sqlmock has no way to make it fail independently of `Columns()`, which is checked first and would already have failed for the same underlying reason (closed rows).

### AD-038: M16 (MySQL/MariaDB adapter) shipped; found `stmt.Insert`/`stmt.Update` needed `PrimaryKey`, and `assignFieldValue` needed numeric→bool (2026-07-05)

**Decision:** `driver/mysql` (`github.com/go-sql-driver/mysql` + `database/sql`, per user decision) mirrors `driver/postgres`'s file layout (`mysql.go`/`connector.go`/`dsn.go`/`dialect.go` + tests). Two core (dialect-agnostic) changes came out of building it:
1. `internal/stmt.Insert` and `internal/stmt.Update` both gained a `PrimaryKey []string` field, populated by `repository.Insert`/`repository.Update` from `r.meta.PrimaryKey`. `driver/postgres` ignores both (its single-round-trip `RETURNING *` never needed PK info) — zero behavior change there. `driver/mysql`'s `Insert` needs it to resolve which columns are the PK (composite/explicit values reused from `Values`; the one omitted auto-increment column resolved via `LastInsertId()`) before reading the row back by PK. `driver/mysql`'s `Update` needs it for a subtler reason: capturing matching rows' PK values *before* running the `UPDATE`, then reading them back by PK afterward — re-running the original `Where` verbatim after the update breaks whenever `Sets` modifies a column `Where` itself filters on (e.g. `Set(&t.Category, "hardware")` after `Where(op.Eq(&t.Category, "tools"))` — a real scenario the M15 conformance suite's `CRUD` group hit immediately against real MySQL). `Update` falls back to re-running `Where` verbatim when `PrimaryKey` is empty (safe for every other current caller of `Dialect.Update` — `SaveOne`/cascade `SetNull`/`Delete`/`Restore` all either never modify a column their own `Where` depends on, or discard the returned rows entirely).
2. `repository.assignFieldValue` gained a numeric→bool conversion rule (`numericToBool`, "!= 0"), both for plain `bool` fields and inside the existing `*X` pointer-wrap branch (AD-037). MySQL has no native boolean type — `BOOLEAN`/`BOOL` are aliases for `TINYINT(1)`, and `go-sql-driver` reports that as a plain `int64`, which Go's `reflect.ConvertibleTo` never bridges to `bool` (unlike Postgres, where the driver already reports a native Go `bool` for a `BOOLEAN` column).
**Reason:** Both gaps are exactly what M15/M16 were built to surface (AD-034's stated goal) — no adapter before MySQL needed a multi-round-trip write path or a numeric-shaped boolean, so nothing in M1-M15 ever exercised either branch.
**Trade-off:** None significant for either change — both are backward-compatible additions (a new struct field ignored by existing callers; a new reflect-based conversion rule that only activates when the exact-type and pointer-wrap checks already failed, so it can't shadow any previously-passing case).
**Impact:** `driver/mysql` fully implemented and passing `internal/dialecttest.Run` against real MySQL 8 (`task test-integration`). `repository/repository_test.go` gained `TestAssignFieldValue_BoolField_NumericRaw`, `TestAssignFieldValue_PointerBoolField_NumericRaw`, `TestNumericToBool`. `driver/mysql/dialect_test.go` gained the PK-read-back `Update` test set. `.docker/docker-compose.test.yml`/`Taskfile.yml` updated (AD-039 covers the DDL-shape and row-normalization side of M16 specifically). Full spec/design/tasks: `.specs/features/mysql-adapter/`.

### AD-036: M15 built as `internal/dialecttest` — adapter supplies DDL, harness owns the Go-side schema (2026-07-05)

**Decision:** `internal/dialecttest` (internal-only, per user decision — every adapter through M21 lives in this same module) exposes `Run(t, schema Schema, caps Capabilities, opts ...golem.Option)`. `Schema` holds 6 dialect-supplied `CREATE TEMPORARY TABLE` DDL strings (`Widget`/`Deleted`/`Parent`/`CascadeChild`/`SetNullChild`/`RestrictChild`, all `conf_`-prefixed to avoid colliding with anything else in the same test database); the harness itself owns the Go-side `entity.New` mappings for those exact tables (`entities.go`) and never generates DDL. `Capabilities.Locking` lets a dialect mark any of the 6 lock strength/wait fields unsupported, causing that one `Locking` subtest to `t.Skip` rather than fail or vanish silently. 9 subtest groups total (`BindScanRoundTrip`, `CRUD`, `SoftDelete`, `Cascade`, `Joins`, `Preload`, `Aggregates`, `Locking`, `ConflictDetection`), each its own `t.Run`. `driver/postgres/conformance_integration_test.go` is the harness's first (and, until M16, only) caller.
**Reason:** Continues AD-034's reasoning directly — this is that milestone, built. The DDL-ownership split (harness defines shape in Go, adapter supplies its dialect's DDL text) exists so the harness itself never needs schema-generation logic, keeping AD-012's "golem never generates DDL" line bright even in test infrastructure; requiring ~6 short `CREATE TEMPORARY TABLE` statements per adapter is a small, one-time cost that also proves the adapter's basic DDL syntax works at all. `Cascade` needing 3 separate physical child tables (not the single `Child` table the first design draft assumed) was discovered mid-implementation — `OnDelete` behavior is fixed per `entity.Entity[Child]` Go mapping (via the FK registry `entity.New` populates at init), not per-row, so exercising all 3 modes genuinely needs 3 separate table+entity pairs; matches `repository/repository_test.go`'s already-proven `cascade_child_delete`/`_setnull`/`_restrict` pattern. `Run`'s signature took `opts ...golem.Option` instead of a `golem.Connector` directly for the same reason: `postgres.New(...)` (and every future adapter's constructor, by the established pattern) returns `golem.Option`, not a raw `golem.Connector` — there's no exported way to get one directly.
**Trade-off:** Building the harness first (M15) instead of jumping straight to M16 has a real upfront cost (a whole package, own schema, own entities) before any second adapter exists to justify it — accepted per AD-034's original reasoning, and paid off immediately (see AD-037).
**Impact:** New package `internal/dialecttest` (10 files: `dialecttest.go`, `entities.go`, `crud.go`, `softdelete.go`, `cascade.go`, `joins.go`, `preload.go`, `aggregates.go`, `locking.go`, `conflict.go` — all `<name>.go`, no naming-convention violations). New `driver/postgres/conformance_integration_test.go`. `.specs/features/cross-dialect-conformance-suite/{spec,design,tasks}.md` record the full requirement/design/task trail, including the schema-shape correction above. Verified green via `task test-integration` (real Postgres via Docker) — both `TestPostgres_Conformance` and the pre-existing `.examples/postgres-minimal-blog` suite pass. CI still doesn't run any integration tests (`task test-integration` requires Docker, never wired into `.github/workflows/ci.yml`) — flagged as a Todo under M15 in ROADMAP.md, since without it this (and every future adapter's) conformance proof only ever runs on whoever's machine happens to have Docker.

### AD-037: M15's first real run immediately found 3 pgx-type-leak bugs in `driver/postgres`/`repository` (2026-07-05)

**Decision:** Fixed all 3 in the same session rather than routing around them in the conformance test:
1. `driver/postgres/dialect.go` gained `normalizeRow`/`normalizeRows`, applied to every `map[string]any` row `Query`/`ExecRaw`/`Insert`/`Update` return: converts `pgtype.Numeric` (any plain, non-CAST `DECIMAL`/`NUMERIC` read) to `float64`, `pgtype.Time` (any `TIME` read) to `time.Time`, `[16]byte` (any `UUID` read) to its formatted string, and a `map[string]any`/`[]any` JSON root (any `JSON`/`JSONB` read) to its marshaled string — all via `reflect.ConvertibleTo`-incompatible pgx-native types that were leaking straight from `pgx.RowToMap`'s default decoding past the `Dialect` boundary into dialect-agnostic `repository` code.
2. `repository/repository.go`'s `assignFieldValue` now wraps a raw non-pointer value in a new `*X` when the target field is `*X` and the raw value's type is `X` or convertible to it — previously only handled the exact-type-match and same-non-pointer-`ConvertibleTo` cases, so any nullable (`*time.Time`-shaped) field reading back a non-NULL row (e.g. `Restore`+`FindOne(...).WithDeleted()` on a soft-deleted-then-restored... no — simpler: any `*T` field whose column is non-NULL for the row being scanned) failed with a `reflect`-inconvertible error.
**Reason:** `internal/dialecttest`'s `BindScanRoundTrip`/`CRUD`/`SoftDelete` subtests are the first tests in this project's history to exercise a plain (non-`SUM`/`AVG`-CASTed) read of a `DECIMAL` column, a `TIME` column read back through `FindOne` (not just bound on insert), a `UUID` read back through `FindOne`, a `JSON` object/array read back through `FindOne`, and a nullable `*time.Time` field reading a non-NULL value through `FindOne(...).WithDeleted())` — every prior test either used a mocked dialect (unit tests, where these pgx-native decode quirks don't exist) or never happened to read one of these column kinds back through a real query at all. This is exactly the value M15 was built for (AD-034's stated goal): catching gaps hand-copied ad hoc tests missed.
**Trade-off:** None significant — these are correctness fixes with no behavior anyone could have been relying on (the old behavior was simply an error on read). `golem.Dialect.Bind`/`Scan` were also confirmed, while investigating this, to be dead code in the current architecture — no production path calls them (`Insert`/`Update` pass Go values straight to pgx as query args, and `Query`/`ExecRaw`/`Insert`/`Update` all decode rows via `pgx.RowToMap` + `repository.assignFieldValue`'s generic reflect conversion, never `Dialect.Scan`). Left as-is (still unit-tested directly, still documented in README as the adapter contract) — flagged as a Deferred Idea below since reconciling documented-contract-vs-actual-code-path is a real but separate cleanup, not blocking M15.
**Impact:** `driver/postgres/dialect_test.go` gained `TestDialect_Query_CollectRowsError`, `TestScan_DECIMAL_PgtypeNumericRaw`, and a `normalizeRow`/`normalizeRows` test block (8 new tests). `repository/repository_test.go` gained 3 new `TestAssignFieldValue_PointerField_*` tests. Coverage held at 100% except `Scan`'s `Float64Value()` error branch (accepted-unreachable, same class as `compilePredicate`'s `default:` — `strconv.ParseFloat` failing on a well-formed digit+exponent string isn't realistically constructible). `internal/dialecttest`'s own `runCRUD` needed 2 of its own test widgets to always populate `UID`/`Meta` with valid (not zero-value) strings — unrelated to the 3 bugs above, just `SaveOne`'s documented whole-row-including-zeros semantics meeting a `UUID`/`JSON` column's stricter-than-`NULL` validation of an empty string.

### AD-035: `golem.GetDataSource(optionalName ...string)` — process-wide DataSource registry, duplicate names error (2026-07-05)

**Decision:** `NewDataSource` now registers the built `*DataSource` in a package-level `map[string]*DataSource` (guarded by a `sync.RWMutex`) keyed by its name (see `DataSourceName`, defaults to `"default"`). `GetDataSource(optionalName ...string)` looks it up by name (first variadic element only, the usual Go idiom for a single optional arg; defaults to `"default"`), returning `ErrDataSourceNotFound` (wrapped with the name) on a miss. `Close()` deregisters the instance (only if the registry still holds that exact pointer under that name), freeing the name for reuse. `NewDataSource` returns a plain error (not a panic) if the name is already registered and not yet `Close()`'d.
**Reason:** User had always intended to support multiple named `DataSource` instances (hence `DataSourceName` existing since M1) but never added a way to retrieve one from outside the call site that built it. Asked directly for `golem.GetDataSource(optionalName ...string)`. On the duplicate-name question (overwrite silently vs. error), user chose to error — consistent with this project's established bias toward loud failures over silent footguns (AD-030's locking-outside-a-transaction guard is the precedent).
**Trade-off:** Erroring on duplicate names meant every test in the `golem` and `repository` packages that built more than one `DataSource`/`DataSource`-backed `Conn` under the same implicit `"default"` name (~30 call sites total) needed a unique `DataSourceName` (usually `t.Name()`, plus a counter where one test built more than one) and a `Close()`/`t.Cleanup` to free it — otherwise the second `NewDataSource` call in the same test process fails with "already registered." Chose this over the simpler "just overwrite" alternative because overwriting silently orphans the previous instance's registry slot while it's still functionally alive, which is exactly the kind of surprising-at-a-distance behavior this project avoids elsewhere.
**Impact:** `data_source.go` gains `dataSourceRegistry`/`dataSourceRegistryMu` and `GetDataSource`; `errors.go` gains `ErrDataSourceNotFound`. `data_source_test.go` gained a `newTestDataSource(t, opts...)` helper (unique name per test + auto-`Close`) used by most of its tests. `repository/repository_test.go`'s shared `newConnWithDialect` helper now takes a counter-suffixed unique name + `t.Cleanup`; its 5 non-helper `golem.NewDataSource` call sites got the same treatment. The two integration-tagged files (`.examples/postgres-minimal-blog/main_integration_test.go`, `driver/postgres/connector_integration_test.go`) got `golem.DataSourceName(t.Name())` added to all 15 of their `NewDataSource` calls for the same reason, since they run sequentially in one process per package and previously all shared the implicit `"default"` name.

### AD-034: ROADMAP.md gets M15-M21 — a conformance suite, then one milestone per INSIGHT.md database (2026-07-05)

**Decision:** Promoted "additional adapters beyond Postgres" from a single Future Considerations bullet into 7 concrete PLANNED milestones: M15 (cross-dialect conformance test harness, no new adapter — extracts `driver/postgres`'s integration-test contracts into a reusable form), then M16 MySQL/MariaDB, M17 SQLite, M18 MSSQL, M19 Oracle, M20 IBM Db2, M21 Snowflake (OLAP, explicitly reduced scope). Ordering is impact/effort: MySQL and SQLite are closest to what `driver/postgres` already assumes (ANSI-ish SQL, real row locking, `RETURNING`-equivalent or cheap two-round-trip inserts); MSSQL/Oracle/Db2 progressively diverge more (table-hint locking, `MERGE INTO`-only upsert, identifier-length limits); Snowflake is last because it's not a transactional OLTP database at all — no row locking, no `CHECK`, and cascade-delete semantics that may not even be wanted for OLAP schemas.
**Reason:** User asked to define concrete next steps toward supporting "every database possible" per INSIGHT.md, which already lists exact type/DDL/pagination/upsert/locking mappings for all 7 (Postgres + 6 more) — that research was sitting unused as a reference doc with no execution plan attached to it. M15 exists as its own milestone (not folded into M16) because building 6 adapters' integration tests by hand-copying `driver/postgres`'s would be enormous, error-prone, and drift over time; a shared harness makes every subsequent adapter milestone smaller and keeps them honest against the same contract `driver/postgres` already proved out (AD-015/AD-016's dialect-agnostic `stmt`/`ColumnType` design was built with exactly this in mind).
**Trade-off:** None of M15-M21 has been started or even lightly prototyped — the milestone descriptions in ROADMAP.md are informed by INSIGHT.md's research and this codebase's existing contracts, not by having actually built against any of these 6 databases yet. Expect scope corrections once M15 (or M16) is actually implemented — e.g. `stmt.LockClause`'s `Strength`/`Wait` strings may need new values for a locking style Postgres/MySQL/Oracle don't share.
**Impact:** `.specs/project/ROADMAP.md` M15-M21 added. `PROJECT.md`'s Constraints (AD-033) already commits this project to `0.x.y` forever, so none of these adapters will ever force a `/v2` import-path change regardless of how many ship. Oracle's identifier-length decision (already flagged in Deferred Ideas below) must be resolved before M19 starts, not during.

### AD-033: Versioning stays on major `0` forever; `0.x.y` simulates major.minor.patch (2026-07-05)

**Decision:** Git tags/GitHub releases for this repo will never cross `v1.0.0`. The version scheme is `0.MAJOR.MINOR` (e.g. `0.1.0` → `0.2.0` for a breaking/feature bump, `0.1.1` for a patch) — the leading `0` is permanently fixed, and what would normally be "major" and "minor" both live inside the digits after it.
**Reason:** User flagged that Go's module system requires any module reaching major version 2+ to carry a `/v2`, `/v3`, ... suffix on its import path (`golang.org/x/mod`'s Semantic Import Versioning rule). That would force `import "github.com/leandroluk/golem"` to become `github.com/leandroluk/golem/v2` on the first breaking change past `v1`, breaking every consumer's import path — considered worse than never reaching a "1.0" milestone at all. Modules on `v0.x.y` are exempt from this rule (Go treats all of `0.x.y` as pre-1.0/unstable, no path suffix ever required), so staying under `v1` sidesteps the problem permanently.
**Trade-off:** The repo can never signal "stable v1.0 API" via the version number the way most libraries do — `0.x.y` conventionally reads as "still unstable" to outside consumers, even once the API is de facto stable. Accepted: avoiding the `/v2` import-path churn (for every future consumer, forever) outweighs the cosmetic signal of crossing `v1`.
**Impact:** First GitHub release cut as `v0.1.0`. Any future release automation (changelog generation, `gh release create`, etc.) must never bump past major `0` — treat the first digit after `0.` as the de facto major version and the second as minor/patch combined, or split further (`0.1.2` = major 1, patch 2) if finer granularity is needed later. See `PROJECT.md`'s Constraints section for the permanent version of this rule.

### AD-032: `relation.ForeignKeyOptions` trimmed from 9 options to just `OnDelete` (2026-07-04)

**Decision:** Removed `Cascade`/`OnUpdate`/`Deferrable`/`CreateForeignKeyConstraints`/`Lazy`/`Eager`/`Persistence`/`OrphanedRowAction` from `relation.ForeignKeyOptions` — along with their backing types (`CascadeOption`, `OnUpdateAction`, `DeferrableMode`, `OrphanedRowActionMode`) and every `Resolved*` getter for them. Only `OnDelete`/`OnDeleteAction` remain.
**Reason:** User asked directly, while discussing a hypothetical migration/sync CLI (see conversation), whether the options AD-027 had flagged as "accepted but no runtime effect" were worth keeping. Re-examining all 9: every single one except `OnDelete` has zero runtime effect AND no plausible path to ever getting one, given decisions already locked in elsewhere — `Cascade*`/`Persistence`/`OrphanedRowAction` need a navigational relation field golem will never have (AD-001/AD-024); `CreateForeignKeyConstraints`/`Deferrable` need DDL generation golem will never do (AD-012, reinforced by this same conversation concluding a full auto-sync/diff migration tool isn't worth building either); `Eager` was explicitly decided to stay manual-only (AD-028, a real Go-generics wall, not a temporary gap); `OnUpdate` needs a PK-mutating operation that doesn't exist and has no reason to be added. With M12/M13 both shipped without ever needing any of these, "keep as an inert placeholder in case it's useful later" stopped being a reasonable bet — an option that silently does nothing is a footgun (a caller reading `.Persistence(false)` and expecting TypeORM-like behavior gets nothing, no error), worse than not offering it at all.
**Trade-off:** If a genuine future need arises (e.g. golem someday adds DDL generation, or a PK-mutation operation), this exact API would need to be re-added — accepted, same reasoning already applied to M13's dropped `Min`/`Max` aggregates: build it when there's a real caller, not speculatively.
**Impact:** `relation/relation.go` shrinks to ~60 lines (`OnDeleteAction` + `ForeignKeyOptions{onDelete}` + 2 methods). `repository/repository.go`'s cascade logic (`applyDeleteCascades`, M11) is untouched — it only ever read `Options.ResolvedOnDelete()`. `.examples/postgres-minimal-blog/entities.go` needed no changes (already only used `.OnDelete(...)`). README.md's `ForeignKeyOptions` example collapsed from a ~55-line, 9-method chain (with inline docs for each) down to the single real option — also let a whole "AVISO: these don't do anything" disclaimer paragraph be deleted entirely, since there's nothing left to disclaim. `.specs/features/relations/spec.md`/`design.md`/`tasks.md` are left describing the original 9-option shape as-shipped (same precedent as the `FindByID` removal, AD-022, and the `UpdateOne`/`UpdateMany` collapse, AD-031) — ROADMAP.md's M11 section is the living summary updated to match current reality.

### AD-031: `UpdateOne`/`UpdateMany` collapsed into one `Update` method — partially supersedes AD-005/AD-014 (2026-07-03)

**Decision:** `Repository[T].UpdateOne` (removed) and `Repository[T].UpdateMany` (renamed to `Update`) built and executed the byte-identical query — same `Where`/`Set` resolution, same `Dialect.Update` call — differing only in post-processing: `UpdateOne` treated 0 affected rows as `golem.ErrNotFound` and returned a single `T`; `UpdateMany` treated 0 rows as fine and returned `[]T`. There is now just `Update(ctx, criteria func(*T, *query.Update[T])) ([]T, error)`, matching the old `UpdateMany` shape (always `[]T`, zero rows is not an error).
**Reason:** User pointed out the `One`/`Many` split added a naming distinction with no behavioral difference underneath — unlike `Insert` vs `Save*` (AD-005, genuinely different inputs: new entity vs. existing runtime instance vs. no instance/just criteria) or `FindOne` vs `FindMany` (genuinely different semantics: guaranteed-single-match-or-error vs. zero-or-more), `UpdateOne`/`UpdateMany` were the same operation wearing two names.
**Trade-off:** Callers who want "did this actually match a row" now check `len(result) == 0` themselves instead of getting `golem.ErrNotFound` for free — a one-line cost for removing a redundant method.
**Impact:** `Update` joins `Insert`/`FindByID`(removed)/`Delete`/`Restore`/`Count`/`Exists` as an AD-014 naming-convention exception (base name unambiguous, no `One`/`Many` needed) — AD-005's core distinction (`Insert*`/`Save*`/`Update*` as three input shapes) and AD-014's `One`/`Many` convention both still stand for every OTHER method; this AD only carves out `Update` as a second exception alongside `Insert`. `SaveOne`/`SaveMany` are unaffected (their arguments genuinely differ: one `*T` vs. variadic `...*T`, unlike `Update`'s identical criteria callback either way). `repository/repository_test.go`'s old `UpdateMany`-shaped tests (previously 0% coverage — see the coverage-gaps Todo) are now `Update`'s tests, at 100%. README.md, ROADMAP.md, PROJECT.md updated to match; `.specs/features/query-builder-and-update/`, `.specs/features/repository-core-crud/`, `.specs/features/hooks/`'s spec/design/tasks docs are left as the historical execution record they are (same precedent as the `FindByID` removal, AD-022 — those docs describe what was built at the time, not living documentation).

### AD-030: M14 shipped — `query.Query[T]` locking, guarded to require a real `golem.Tx` (2026-07-03)

**Decision:** `query.Query[T]` gains `.ForUpdate(wait ...LockWait)`/`.ForNoKeyUpdate(...)`/`.ForShare(...)`/`.ForKeyShare(...)` (each optionally `LockWaitNoWait`/`LockWaitSkipLocked`). `Repository[T].FindMany`/`FindOne` return an error — before compiling or issuing any query — if a lock is requested and `r.conn` isn't a `golem.Tx`. `repository.Aggregate` gets no locking support at all.
**Reason:** `SELECT ... FOR UPDATE` outside an explicit transaction is valid SQL that does nothing useful (Postgres's implicit per-statement transaction releases the lock before the caller's next statement runs) — a silent footgun exactly like the join-fan-out and unused-`ForeignKey`-target bugs found earlier this session (STATE.md L-003, AD-027's "target was never read" fix). Erroring loudly instead of silently no-op'ing matches this project's established bias. `Aggregate` has no locking support because Postgres itself rejects `FOR UPDATE` combined with `GROUP BY`/aggregate functions — not a golem scoping choice, a hard DB restriction.
**Trade-off:** None significant — the guard only blocks a usage pattern that never worked correctly anyway.
**Impact:** New `stmt.LockClause` on `stmt.Select` (M1/AD-016's asymmetric contract extended once more — still just `CompileSelect`, no new `Dialect` methods). Verified against real Postgres with an actual two-goroutine, two-transaction, channel-synchronized blocking test (`TestBlogExample_PessimisticLocking_ForUpdateBlocksConcurrentLocker`) — not just SQL-shape unit assertions, since blocking is a runtime concurrency property no amount of string-matching can prove. Every new function at 100% coverage (AD-026). This closes ROADMAP.md's M1-M14 — the full milestone list as of this writing has nothing left "planned"; see Deferred Ideas below for what could come next. Full reasoning: `.specs/features/pessimistic-locking/design.md`.

### AD-029: M13 shipped — `repository.Aggregate[T, R]`, `Min`/`Max` deliberately dropped (2026-07-03)

**Decision:** `repository.Aggregate[T, R any](ctx, r *Repository[T], fn func(t *T, res *R, a *query.Aggregate[T, R])) ([]R, error)` — same free-function-with-2-type-params shape as `Preload`/`join.Inner` (methods can't add a type parameter). `R` is a plain struct, resolved by field-pointer offset against both `T`'s and `R`'s zero values — no `entity.New` needed for `R`, no tags. `GroupBy`/`Sum`/`Avg`/`Count`/`CountAll`/`Where`/`Having`/`OrderBy`/`Limit`/`Offset`/`WithDeleted` on `query.Aggregate[T, R]`. `Min`/`Max` are NOT included.
**Reason:** `Sum`/`Avg` need a `CAST(... AS DOUBLE PRECISION)` in the generated SQL so pgx never hands back `pgtype.Numeric` for an integer column's aggregate (Postgres promotes `SUM`/`AVG` of integer columns to `NUMERIC`) — but that same blanket cast would be wrong for `MIN`/`MAX`, which are valid over non-numeric columns (text, timestamps) where casting to a float makes no sense. Supporting `MIN`/`MAX` correctly needs per-column-type-aware casting, which is real, separate scope beyond what ROADMAP.md's M13 goal (`GroupBy`/`Sum`/`Avg`/`Having`) actually called for.
**Trade-off:** Callers wanting `MIN`/`MAX` must drop to raw SQL (`Repository[T].Exec`, M9) for now.
**Impact:** New `stmt.Projection`/`stmt.Select.GroupBy`/`.Having`/`stmt.AggregateComparison` (internal AST, M1/AD-016's asymmetric contract extended again). `HAVING` clauses always repeat the full aggregate expression (`HAVING CAST(SUM(...) AS DOUBLE PRECISION)>$N`), never the SQL alias — Postgres doesn't allow referencing SELECT-list aliases in `HAVING` (unlike `ORDER BY`, which does, and which `Aggregate`'s `OrderBy` relies on). `scanRow`'s field-assignment logic was extracted into a shared `assignFieldValue` helper, reused by `Aggregate` (pure refactor, no behavior change). Every new function at 100% coverage (AD-026). Full reasoning: `.specs/features/aggregations/design.md`.

### AD-028: M12 shipped — `repository.Preload`, no `Eager` auto-wiring (a real Go-generics wall, not a choice) (2026-07-03)

**Decision:** `repository.Preload[T, J any](ctx, r *Repository[T], items []T, target *entity.Entity[J], criteria ...) (map[any][]J, error)` — a free function (mirrors `join.Inner`'s two-type-param shape), reusing M11's FK registry to auto-discover the join column in either direction (works whether `target` declares the FK or `T`'s own entity does). `ForeignKeyOptions.Eager(true)` (accepted since M11) stays metadata-only — NOT wired to automatically run inside `FindMany`/`FindOne`.
**Reason:** Auto-wiring `Eager` hit a genuine wall, not a preference: `FindMany`'s signature (`([]T, error)`) is fixed and used everywhere; the related type varies per FK (2+ `Eager` FKs on one entity could point at 2+ different types), so there's no way to return typed preloaded data through that signature without either (a) attaching to `T` (ruled out, AD-001/AD-024) or (b) a stateful side-channel method (rejected — not idiomatic, racy under concurrent reuse of one `Repository[T]`).
**Trade-off:** Less "automatic-feeling" than TypeORM's `eager: true`. Accepted: two explicit lines (`FindMany` then `Preload`) beats hidden magic, matches this project's existing bias.
**Impact:** `repository/preload.go` (new). Every new function at 100% coverage (AD-026). `.examples/postgres-minimal-blog` gets `TestBlogExample_Preload_LoadsPostsPerUser`. Full reasoning: `.specs/features/preload-eager-loading/design.md`.

### AD-027: M11 shipped — only `OnDelete` gets real cascade behavior, via a global FK registry (2026-07-03)

**Decision:** `entity.Table.ForeignKey` now takes a 3rd variadic `...*relation.ForeignKeyOptions`. Of the 9 documented options, only `OnDelete` (`Cascade`/`SetNull`/`Restrict`) changes `Repository[T]` behavior — implemented via a new package-level registry (`entity.ForeignKeysReferencing`, populated as a side effect of `entity.New` running `ForeignKey`, keyed by the TARGET/parent table since the declaration direction is child→parent, the opposite of what cascade needs to query) consulted by `Repository[T].Delete`, which wraps cascade+delete in an implicit transaction (reusing an existing `Tx` if already inside one) only when 1+ registered FK is cascade-actionable.
**Reason:** This was flagged to the user mid-design (see conversation) before implementation: `Cascade*`/`Persistence`/`OrphanedRowAction` (TypeORM semantics) all require an attached in-memory related object on the owning struct to cascade a write from/to — which AD-001/AD-024 already ruled out (no navigational collection field). `CreateForeignKeyConstraints`/`Deferrable` describe DDL-time behavior golem never emits (AD-012). `OnDelete`/`OnUpdate` are the only two that operate purely on "who references this row," needing no attached object — and only `OnDelete` had an operation (`Delete`) to hang real behavior off of; no operation currently mutates a PK value, so `OnUpdate` has nothing to trigger from yet.
**Trade-off:** 5 of 9 options (`Cascade*`, `Persistence`, `OrphanedRowAction`, `CreateForeignKeyConstraints`, `Deferrable`) are accepted/stored but functionally inert — a caller setting them gets no error but also no behavior, which could surprise someone coming from TypeORM. Mitigated by loud documentation (package doc in `relation/relation.go`, README.md inline comments, spec.md/design.md) rather than silently rejecting them, so the API shape stays a faithful superset of the original design and nothing needs to change if a future milestone finds a way to make more of them real (e.g. once M12's Preload exists, `Eager` gets wired up for real, no API break).
**Impact:** Also fixed a pre-existing bug found while implementing this: `ForeignKey`'s `target` argument was NEVER type-asserted or read before M11 — `ForeignKeyMeta` only stored `FieldName`. Now stores `ColumnName`/`TargetTableName`/`TargetPrimaryKey`/`Options` too, and panics if `target` doesn't implement `Describe() EntityMeta` or has a composite PK (composite-PK FK targets unsupported — `ForeignKey` only resolves one `fieldPtr`). New packages/functions from this milestone (`relation`, `entity.FKRegistration`+registry, `repository`'s `cascadeActionable`/`beginCascadeTx`/`applyDeleteCascades`/`countValue`) are all at 100% statement coverage per AD-026. Full design: `.specs/features/relations/design.md`.

### AD-024: `Preload`/`With` stays a separate query-level API, not a struct field — AD-001 upheld (2026-07-03)

**Decision:** When M12 (Preload/Eager Loading) is built, related rows load into a separate structure (e.g. `map[ParentID][]Child`, or a typed paired-result returned alongside `T`), never by adding a navigational-collection field (`Posts []Post`) onto the entity struct itself. `ForeignKeyOptions.Eager(true)` (M11) means "run the same Preload/With mechanism automatically when the query doesn't call it explicitly" — it does not imply or require a struct field to populate.
**Reason:** User was asked directly ("keep AD-001, separate API" vs. "revoke AD-001, add a relation field to the struct") and deferred to whichever is most coherent. Keeping AD-001 avoids a breaking change to every already-declared entity struct and preserves the project's core "plain struct, zero magic" positioning — a struct field would need either a pointer/slice added to every entity that might ever be eager-loaded (viral, breaks existing structs) or reflection-based dynamic field injection (far more magic than the project has ever done elsewhere).
**Trade-off:** Ergonomics are a step below TypeORM's `user.posts` direct property access — callers get a side-table/paired-result instead of just reading a field off the returned `T`. Accepted: matches AD-001's original trade-off acceptance almost verbatim.
**Impact:** M12's design.md must pin down the exact `Preload`/`With` return shape before implementation (not yet decided — this AD only fixes *where it doesn't go*, not the final API surface).

### AD-023: Duplicate/FK-violation sentinels wrap the driver error, not replace it (2026-07-03)

**Decision:** `driver/postgres/dialect.go`'s `mapError(err error) error` checks `errors.As(err, &pgconn.PgError)`; on SQLSTATE `23505`/`23503` it returns `fmt.Errorf("%w: %w", golem.ErrDuplicateKey /* or ErrForeignKeyViolation */, err)` (Go 1.20+ multi-`%w`) instead of returning a bare sentinel. Applied at every point `Insert`/`Update`/`Query`/`Exec`/`ExecRaw` in the Postgres dialect surface a driver error.
**Reason:** Callers need both `errors.Is(err, golem.ErrDuplicateKey)` (branch on kind) AND `errors.As(err, &pgErr)` (inspect the real `*pgconn.PgError` — constraint name, detail, etc.) without losing either. A bare sentinel would satisfy the first but break the second.
**Trade-off:** None — this is strictly additive over the plain driver error other than message text.
**Impact:** `errors.go` gains `ErrDuplicateKey`/`ErrForeignKeyViolation`. `IsConflict` (used by the hook system, AD-007/M7) is unrelated and unchanged — it stays a broad "any class-23 violation" check for deciding whether to run `OnConflictCreate`/`OnConflictUpdate`/`OnConflictDelete` hooks, independent of the new typed-sentinel wrapping.

### AD-001: No `ManyToMany`/`JoinTable` relation type (2026-07-02)

**Decision:** Many-to-many relationships are modeled as a plain entity (the junction table) with two `ForeignKey`s — no dedicated relation type or `@JoinTable`-style API.
**Reason:** A junction table is exactly what the database does under the hood; hiding it behind a parallel API (like TypeORM's `@JoinTable`/`@JoinColumn`) just adds a second way to describe the same thing.
**Trade-off:** No automatic navigational collection (`question.Categories []Category` populated for free). Loading that becomes a query-level concern later (see Future Considerations in ROADMAP.md — `Preload`/`With`).
**Impact:** `entity.Table` never needs a `ManyToMany` method. `ForeignKey` covers every relation shape (one-to-one, one-to-many/many-to-one, and many-to-many via the junction entity).

### AD-002: Cyclic entity references resolved via thunk, not two-phase declaration (2026-07-02)

**Decision:** When two entities reference each other (mutual `ForeignKey`/relation), pass `func() *entity.Entity[T] { return X }` instead of `X` directly to break the Go package-init cycle.
**Reason:** The alternative (`entity.New[T]()` then a separate `.Define(func(...){...})` call) avoids the cycle too, but forces every entity to be declared in two places, which was rejected as less readable.
**Trade-off:** Slightly unusual syntax (thunk) for the rare cyclic case; unnecessary for the common non-cyclic case (pass the entity var directly, e.g. `ForeignKey(&t.OwnerUserID, UserEntity)`).
**Impact:** Only needed when entity A's builder closure references entity B and vice versa. In practice this became mostly moot after AD-001 (junction-entity pattern has no cycles: the junction references both sides, neither side references the junction back).

### AD-003: No generic `Manager` — only `Repository[T]` (2026-07-02)

**Decision:** Dropped the TypeORM-style `dataSource.manager` (entity-agnostic manager). Every CRUD/query operation goes through `Repository[T]`, always bound to one entity type.
**Reason:** A parallel "generic manager vs typed repository" API doubles the surface for no real benefit — `repository.Get(conn, Entity)` is cheap to call per entity type.
**Trade-off:** Saving multiple different entity types "in one call" (e.g. `manager.save(a, b, c)` across types) now takes one `repository.Get(...).Insert*` call per type instead of one combined call. Accepted as a rare case.
**Impact:** `Transaction` moved from `Manager` to `DataSource` directly (`dataSource.Transaction(ctx, func(tx golem.Tx) error {...})`).

### AD-004: `golem.Conn` unifies `*DataSource` and `golem.Tx` (2026-07-02)

**Decision:** `repository.Get(conn golem.Conn, e *entity.Entity[T])` and `golem.Conn.Exec` accept either `*golem.DataSource` or `golem.Tx` — both implement the same interface.
**Reason:** Without this, every call site (including inside hooks and inside `Transaction` callbacks) would need to pass both the datasource AND the tx separately, which was flagged as repetitive.
**Trade-off:** None significant — this is a pure simplification.
**Impact:** Hooks only need `conn golem.Conn` (not a wrapper `entity.Context` type, see AD-007). Inside `dataSource.Transaction`, `repository.Get(tx, Entity)` replaces `repository.Get(dataSource, Entity, tx)`.

### AD-005: Three distinct write concepts — `Insert*` / `Save*` / `Update*` (2026-07-02)

**Decision:** `Insert`/`InsertMany` = brand-new entity (no runtime instance yet). `SaveOne`/`SaveMany` = re-persist an instance you already have in runtime (e.g. from a prior `Insert`/`FindByID`). `UpdateOne`/`UpdateMany` = criteria-based `Where`+`Set` update with no instance at all.
**Reason:** These are genuinely different operations with different inputs (new data vs. existing instance vs. no instance/just criteria); collapsing them into one `Save`/`Update` was ambiguous about what each call needed.
**Trade-off:** More methods on `Repository[T]` (8 write methods instead of 2-3), but each name states its precondition unambiguously.
**Impact:** Naming rule going forward: suffix `One` always returns a single `T`, suffix `Many` always returns `[]T`. `Insert`/`FindByID` are the two exceptions that omit `One` because the base name is already unambiguous.

### AD-006: Fluent hook builder, no per-hook wrapper types (2026-07-02)

**Decision:** `entity.AddHook(Entity).BeforeCreate(fn).AfterCreate(fn)...` instead of `entity.AddHook(Entity, entity.BeforeCreateHook(fn))`.
**Reason:** Matches the builder-chain style already used elsewhere (`ForeignKeyOptions`, `JoinTableOptions` in early drafts); removes the need for a dedicated exported type (`BeforeCreateHook`, `AfterCreateHook`, ...) per hook slot.
**Trade-off:** None identified.
**Impact:** Registering the same slot twice on the same entity must panic (naming the slot) — noted as a requirement for M7, not yet a design detail (error message format TBD).

### AD-007: Hooks take `(ctx, *T, golem.Conn)` directly — no `entity.Context` wrapper (2026-07-02)

**Decision:** Hook signature is `func(ctx context.Context, i *T, conn golem.Conn) error`. An earlier `entity.Context` interface (wrapping `Context()`/`Tx()`/later `DataSource()`) was designed and then removed.
**Reason:** Once `golem.Conn` existed (AD-004), the wrapper added no value — `conn` alone is enough to build a `repository.Get(conn, OtherEntity)` inside a hook, and `ctx` is idiomatically a separate, explicit parameter (matches every other method in the API, e.g. `repo.Save(ctx, ...)`).
**Trade-off:** None — straightforward simplification.
**Impact:** One less exported type in `entity` package.

### AD-008: `Unique`/`Index` live on `Builder` (table scope), not chained on `Col` (2026-07-02)

**Decision:** `b.Unique(&t.Email)` and `b.Index(&t.OwnerUserID)` are `entity.Table` methods (like `PrimaryKey`), not `.Unique()`/`.Index()` chained off `Col(...)`.
**Reason:** Both can be composite (span more than one column), so they can't be scoped to a single `Col` call.
**Trade-off:** None.
**Impact:** `entity.Column` (the `Col` chain) only has single-column concerns: `.Name`, `.Nullable`, `.Default`, `.DefaultFunc`.

### AD-009: `CreateDate`/`UpdateDate` always auto-fill — no `.AutoNow()` toggle (2026-07-02)

**Decision:** Removed `.AutoNow()`. Marking a field via `CreateDate`/`UpdateDate` always means "fill this with the operation's timestamp automatically."
**Reason:** There's no realistic case where you'd mark a field as "created at" and NOT want it auto-filled — the toggle was dead weight, always called together with the marker in every example.
**Trade-off:** None.
**Impact:** `b.CreateDate(&t.CreatedAt)` / `b.UpdateDate(&t.UpdatedAt)`, full stop.

### AD-010: ~~Column types live on the dialect package (`postgres.BIGINT()`, etc.)~~ — SUPERSEDED by AD-015 (2026-07-02, superseded 2026-07-03)

**Decision:** No separate generic `conector`/`connector` types package — `postgres.BIGINT()`, `postgres.VARCHAR(50)`, etc. live directly on the adapter.
**Reason:** Early drafts referenced an undefined `conector.*` package inconsistently; consolidating on the adapter package removed the dangling import and the "which package has the types" ambiguity.
**Trade-off:** Column type constructors aren't portable across dialects without a rewrite if a second adapter (MySQL, etc.) is added later — accepted, revisit only if/when a second adapter is actually built.
**Impact:** Every entity example imports `postgres "github.com/leandroluk/golem/driver/postgres"` for both connection options AND column types.
**Why superseded:** user decided upfront (before M2 shipped any code) to support multiple dialects (MySQL, MSSQL, Oracle) properly instead of waiting for "if/when a second adapter is actually built" — see AD-015.

### AD-011: `query.Count[T]`, `query.Join[T]`, `query.Update[T]` are distinct from `query.Query[T]` (2026-07-02)

**Decision:** Each write/read shape gets its own criteria type instead of reusing `query.Query[T]` (which has `Select`/`Where`/`OrderBy`/`Limit`/`Offset`) for everything.
**Reason:** `Count`/`Exists` only need `Where`. `Update` needs `Where`+`Set`. `Join` needs `On` (column-to-column) plus its own `Where` (column-to-value) — reusing `op.Eq` for both column-to-column and column-to-value comparisons would be ambiguous, so `Join` got a dedicated `.On(fieldPtr, fieldPtr)`.
**Trade-off:** More types to document, but each is minimal and matches exactly what that operation needs.
**Impact:** `op.*` (e.g. `op.Eq`, `op.Gte`) is always "column vs. literal value"; `.On(...)` is always "column vs. column." Never conflate the two.

### AD-012: Soft delete filters by default; Migrations explicitly out of scope (2026-07-02)

**Decision:** Any entity with `DeleteDate` declared gets automatic `WHERE deleted_at IS NULL`-style filtering on every `Where`-capable builder (`Query`/`Count`/`Update`/`Join`), opt-out via `.WithDeleted()`. Separately, schema migrations/synchronization were declared out of scope for `golem` entirely.
**Reason:** Matches standard ORM soft-delete behavior (TypeORM does the same). Migrations were cut because the user already uses Liquibase externally and successfully — building a competing migration system is large, separate scope with no clear benefit over an existing mature tool.
**Trade-off:** None for soft-delete. For migrations: `golem` entities are runtime mapping only, never a DDL source of truth — anyone using this module needs a separate schema-management tool.
**Impact:** `Delete()`/`Restore()` on `Repository[T]` exist specifically to pair with `DeleteDate`. No `Synchronize()`/migration-runner will ever be built into this module.

### AD-013: `golem.Conn.Exec` returns `golem.Result`, not `[]map[string]any` (2026-07-02)

**Decision:** Raw SQL execution (`Exec`) returns a `golem.Result` interface (`Next() bool`, `Scan() (map[string]any, error)`, `RowsAffected() (int64, error)`), not a plain `[]map[string]any`.
**Reason:** Different statements return fundamentally different things — a `SELECT` has rows to iterate, a plain `UPDATE` (no `RETURNING`) only has an affected-row count, and `UPDATE ... RETURNING` has both. A single `[]map[string]any` return can't represent the "just a count, no rows" case cleanly.
**Trade-off:** One more type to learn (`golem.Result`) vs. a bare slice, but it mirrors the already-familiar `database/sql.Rows` iteration pattern.
**Impact:** `Repository[T].Exec` is unaffected — it always has a known destination type `T`, so it returns `([]T, error)` directly, no `Result` involved.

### AD-014: `FindMany`/`FindOne` naming, `One`/`Many` suffix convention (2026-07-02)

**Decision:** Settled on `FindMany` (not `Find` or `FindAll`) paired with `FindOne`. General rule going forward: any method ending in `One` returns a single `T`; any method ending in `Many` returns `[]T`.
**Reason:** `FindAll`/`SaveMany` naming was inconsistent early on (mixed with a bare `Find`); `Many` was chosen over `All` because these methods operate on exactly the entities/criteria you passed, not literally "the whole table" (calling it `All` when it might return one row would be misleading).
**Trade-off:** None.
**Impact:** Applies to `InsertMany`, `SaveOne`/`SaveMany`, `UpdateOne`/`UpdateMany`, `FindOne`/`FindMany`. `Insert`, `FindByID`, `Delete`, `Restore`, `Count`, `Exists` are the accepted exceptions (base name already unambiguous about cardinality).

### AD-015: Dialect-agnostic `golem.ColumnType` + `golem.Dialect` contract (2026-07-03, supersedes AD-010)

**Decision:** Column type constructors (`golem.BIGINT()`, `golem.VARCHAR(50)`, `golem.TEXT()`, `golem.BOOLEAN()`, `golem.TIMESTAMPTZ()`, `golem.UUID()`, `golem.JSON()`, etc.) live in the core `golem` package, not on the adapter. Each adapter (`postgres`, and future `mysql`/`mssql`/`oracle`) implements a `golem.Dialect` interface (`Bind(t ColumnType, value any) (driver.Value, error)`, `Scan(t ColumnType, raw any, dest any) error`) that knows how to marshal/unmarshal each semantic type for that specific driver.
**Reason:** User wants real multi-database support (MySQL, MSSQL, Oracle) to be architecturally possible from the start, not deferred until "a second adapter is actually built" (AD-010's original stance). Since `golem` never generates DDL (migrations are out of scope, AD-012), the column type was never really about schema — it's about telling the adapter how to bind/scan values for types `database/sql` can't infer alone (UUID, JSON/JSONB, arrays, ENUMs, etc.), and that lookup can be keyed by a dialect-agnostic semantic type just as well as a dialect-specific one.
**Trade-off:** Adding a new adapter is still real work (SQL generation differs a lot per dialect — placeholders, pagination, upsert, `RETURNING` vs `OUTPUT` vs `RETURNING INTO`), but the type system itself and every entity declaration are portable across adapters without changes. MySQL/SQLite are "moderate effort" adapters (closer to ANSI SQL); MSSQL/Oracle are "high effort" (syntax diverges more).
**Impact:** Every `Col(fieldPtr, type)` call in `entity.Table` now takes `golem.ColumnType` instead of an adapter-specific type. All README examples updated (`postgres.BIGINT()` → `golem.BIGINT()`, etc. — the `postgres` import is still needed for `postgres.New`/`postgres.Options`, just not for column types anymore). A rejected alternative was a separate `golem/type` package — impossible, `type` is a Go reserved keyword and can't be a package name.

**Considered follow-up (not yet decided):** exact initial `golem.ColumnType` set beyond what README examples use (`BIGINT`, `INT`, `VARCHAR`, `TEXT`, `BOOLEAN`, `TIMESTAMPTZ`, `UUID`, `JSON`) — grows on demand (YAGNI), needed before M2 implementation starts.

### AD-016: `internal/stmt` AST + asymmetric `Dialect` statement contract (2026-07-03, extends AD-015)

**Decision:** `golem.Dialect` grows beyond `Bind`/`Scan` to also cover statement generation, via a shared internal AST (`golem/internal/stmt`: `stmt.Select`, `stmt.Insert`, `stmt.Update`, `stmt.Delete`) that `query.Query[T]`/`query.Update[T]`/`query.Count[T]`/`query.Join[T]`/`Repository[T]` build internally and hand to the active `Dialect`. The contract is **asymmetric by statement kind**, not 4 symmetric `Compile*` methods:
  - `CompileSelect(s *stmt.Select) (sql string, args []any, err error)` — pure compile, always 1 round-trip (used by `FindMany`/`FindOne`/`FindByID`/`Count`/`Exists`/joins).
  - `CompileDelete(s *stmt.Delete) (sql string, args []any, err error)` — pure compile, always 1 round-trip, no row data needed (`Delete`/`Restore` only return `error`, never `T`).
  - `Insert(ctx context.Context, conn golem.Conn, s *stmt.Insert) (map[string]any, error)` — **execute**, not compile. The adapter decides internally how many round-trips it needs.
  - `Update(ctx context.Context, conn golem.Conn, s *stmt.Update) ([]map[string]any, error)` — same reasoning as `Insert`.
**Reason:** `RETURNING` (or equivalent) is not universal. Postgres/Oracle/MSSQL have a native one-round-trip way to get the affected row back (`RETURNING`, `RETURNING INTO`, `OUTPUT`). MySQL does not — getting the full row back after an `INSERT`/`UPDATE` (required because `Insert`/`InsertMany`/`SaveOne`/`SaveMany`/`UpdateOne`/`UpdateMany` all return `T`/`[]T` per AD-005) needs a second `SELECT` there. A pure `Compile → (sql, args)` contract can't express "maybe 2 queries," so `Insert`/`Update` must be full `Execute`-style methods where the adapter owns the round-trip strategy. `Select`/`Delete` never need this (no dialect lacks a working `SELECT`, and `Delete`/`Restore` don't return row data at all), so they stay pure-compile.
**Trade-off:** `Dialect` is no longer 4 symmetric methods — slightly more surface to document, but each method's shape now matches what it actually needs to do per dialect instead of forcing every dialect through the same (Postgres-shaped) assumption.
**Impact:** `stmt.*` types are internal (`golem/internal/stmt`), not part of the public API — end users never construct them directly, only `query.*`/`Repository[T]` (via `golem`) and adapters touch them. Side effect: since `Dialect`'s method signatures reference internal types, only adapters built inside the `golem` module tree can implement it — acceptable, since that was already true in practice (nobody outside the module was expected to write a conforming adapter anyway). `stmt.*` grows incrementally per milestone (M1: minimal `Select`/`Insert`/`Update`/`Delete` skeletons with table ref + PK-equality `Where` only — PK-equality is AND of 1+ column=value checks from day one, since composite PKs already exist, e.g. `QuestionToCategory`; M4: full arbitrary predicate tree; M5: `Set` clause on `stmt.Update` for criteria-only updates; M6: `Join` on `stmt.Select`), not fully speced upfront.

### AD-017: `op.Not(condition)` composes over any condition — no dedicated negated variants (2026-07-03)

**Decision:** There is no `op.NotIn`, `op.NotEq`, `op.NotLike`, etc. Negation is always `op.Not(op.In(...))`, `op.Not(op.Eq(...))`, `op.Not(op.Like(...))` — one composable `Not` wrapping any other condition (including `Or(...)`).
**Reason:** `NOT (x IN (list))` and `x NOT IN (list)` are semantically identical in SQL (same for other operators) — enumerating a negated variant for every comparison operator doubles the `op` package surface for zero functional gain.
**Trade-off:** None — this is a pure surface reduction, not a capability trade-off.
**Impact:** `op` package only needs positive comparison operators (`Eq`, `Gt`, `Gte`, `Lt`, `Lte`, `In`, `Like`, ...) plus `Or`/`Not` for composition. The exact positive operator list is still open (see Todos).

## Lessons Learned

### L-003: `.examples/` never actually runs under `go test ./...` — a real bug sat undetected (2026-07-03)

**Context:** While validating M10, ran `.examples/postgres-minimal-blog`'s full integration suite directly (explicit path) instead of via `task test-integration`, and `TestBlogExample_FullFlow` failed: a `join.Inner` query returned 2 rows for 1 user (fan-out from 2 matching posts, no dedup) — a pre-existing bug, unrelated to M10.
**Problem:** Go's `./...` pattern silently skips dot-prefixed directories (`.examples`, `.docker`, `.specs`, etc.) in `go build`/`go test`/`go vet`. `task gate-quick`/`task gate-full`/`task test-integration` all invoke `go test ... ./...`, so `.examples/postgres-minimal-blog`'s tests have *never* run as part of any gate — every milestone from M4 onward was declared "done" partly on the strength of an example test suite that silently never executed.
**Prevents:** Don't trust `task gate-full`/`test-integration` alone to have covered `.examples/` without checking — the `...` wildcard skips dot-prefixed dirs even when spelled out explicitly (`./.examples/...` also matches nothing; confirmed by testing it directly). **Fixed:** `Taskfile.yml`'s `test-integration` now has a second `go test -tags=integration ./.examples/postgres-minimal-blog` step using the exact (non-wildcard) package path, and `repository.FindMany` now dedupes 1:N join fan-out by parent PK (see Todos) — `TestBlogExample_FullFlow` passes for real now.

### L-002: Filenames in this repo are snake_case; the "unauthorized rename" I fought was actually the user's own convention (2026-07-03, corrected same day)

**Context:** During M2/M3 execution, files kept turning up renamed to snake_case (`columntype.go`→`column_type.go`, `datasource.go`→`data_source.go`) while sub-agents worked in the shared working tree. I assumed this was sub-agent hallucination (models sometimes "correct" perceived naming inconsistencies) and reverted it twice, and recorded a now-corrected lesson blaming sub-agents for it.
**Correction:** The user clarified they were the one renaming files themselves (for consistency) while work was in progress, and separately moved `testdata/` + `docker-compose.test.yml` into a new `.docker/` directory. Neither was agent misbehavior.
**Convention going forward:** This repo uses **snake_case filenames** (`column_type.go`, `data_source.go`, `data_source_test.go`, etc.) — NOT the no-underscore style (`columntype.go`) I initially assumed and defended in sub-agent prompts. Docker/test infra lives under `.docker/` (`.docker/docker-compose.test.yml`, `.docker/testdata/schema.sql`), tracked in git (removed from `.gitignore`).
**Prevents:** Don't assume a repo-wide naming convention from a handful of early files without checking for concurrent user edits; when files change unexpectedly mid-session, ask before reverting rather than fighting it twice. The underlying practice from the original (wrong) lesson still stands, though: run `git status --short` after every sub-agent task before staging, to catch anything genuinely unintended (scope creep, stray scratch files) — just don't assume renames are automatically agent-caused.

### L-001: Direct README iteration worked better than upfront brainstorming for this feature (2026-07-02)

**Context:** Started with the `superpowers:brainstorming` flow, but the user redirected almost immediately to editing the design README directly and iterating example-by-example.
**Problem:** A formal spec-first pass would have required guessing at API shapes (`Manager` vs `Repository`, `ManyToMany` vs junction entity, `Save` vs `Insert`/`Update` split) that only became clear by writing concrete code examples and having the user react to them.
**Solution:** Let the README's worked examples be the spec. Each iteration surfaced one concrete design question (naming, cyclic refs, error handling) that was resolved before moving to the next example.
**Prevents:** Don't force a heavyweight spec process when the user is already iterating on a living example document — formalize into `.specs/` once the shape has stabilized (this is what M1-M10 in ROADMAP.md now capture).
